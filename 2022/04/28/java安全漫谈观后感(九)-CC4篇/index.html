TransformingComparator.compare()

通过学习前面的commons-collections链可以知道commons-collections这一系列的关键正是可以触发到各种Transformer的transform()方法，达到任意方法调用的效果。我们之前学习了LazyMap.get或TransformMap.put都可以触发transform()方法，看这个CommonsCollections2的Gadget Chain，不出意外TransformingComparator.compare()应该也可以触发transform()方法。跟进看看

可以发现确实如此，TransformingComparator.compare()可以触发transformer的transform()方法。

我们还可以发现，compare()方法传入的参数obj1,obj2会被当作transform()方法的input值，且一个队列会做出compare行为，很明显是使用queue中的元素做compare操作。通过前面的对构造无数组CommonsCollections链的学习，我们应该敏感地察觉到这里有构造一个新的无数组CommonsCollections链的潜力。

为啥反射改queueArray[0]

看到yso中这个写法我第一反应是因为transform()会分别在queue中的两个元素各触发一次，这样写似乎是为了避免多次触发gadget。

然而我调完上面不使用数组的CommonsCollections2后，我queue.add了两个TemplatesImpl对象obj，但是只弹出来了一次计算器。在transform()方法下断点debug发现了原因:

在触发transform(obj1)时，执行命令弹出计算器后，由于逻辑问题会抛出异常，因此不会再触发transform(obj2)。

即我们gadget中只有queue的第一个元素会触发transform()方法。

所以ysoserial中反射修改的是queue中的第一个元素值，甚至完全不用管第二个元素值是多少都行，因为反正不会再触发transform()。

同理,如果我们尝试在添加队列元素时，只传入一个TemplatesImpl对象

修复

PriorityQueue的利用链是否支持在commons-collections 3中使用？

因为这条利用链中的关键类org.apache.commons.collections4.comparators.TransformingComparator ，在commons-collections4.0以前是版本中是没有实现Serializable接口的，无法在序列化中使用。

Apache Commons Collections官方是如何修复反序列化漏洞的？

  • Commons-Collections 3.2.2增加了一个方法用于检测反序列化是否安全。即默认情况下，反序列化包含常⻅的危险Transformer类(InstantiateTransformer、InvokerTransformer、PrototypeFactory、CloneTransformer等)，会抛出异常

  • Commons-Collections 4.1常见的Transformer类不再实现 Serializable 接口.
